Created: Jan 10, 2025 · 1 min read

Item 5: auto优先于显示类型声明

首先，在C/C++中声明一个变量，如果不进行初始化，其默认值可能会因为其所在的作用域的不同而不同，在使用的时候可能会出现意想不到的错误。而auto类型的变量类型是从初始化进行推导的，所以需要在定义auto变量的时候进行初始化。使用auto变量可以强制对其进行初始化。

int a; //ok
auto x = 10; //ok
auto a; //error: declaration of ‘auto a’ has no initializer

其次，使用auto可以避免冗长的显示类型声明，特别是在操作STL的各种容器及迭代器等, auto可以节省很多时间，如：

std::map<std::string, std::map<int, std::vector<std::string> > m;
auto index = m.find("auto") //虽然可使用typedef，但是还是需要敲冗长的代码

以及一些只有编译器才知道的类型，例如lambda表达式：

auto derefUPLess = [](const std::unique_ptr<Widget>& p1, const std::unique_ptr<Widget>& p2) {
   return *p1 < *p2;
};

再次，编译器更懂变量的数据类型，把一些复杂的工作交给编译器来做是一个不错的选择。auto还可以推导只有编译器才知道数据类型，如lambda表达式构造的闭包，虽然std::function也可以适配lambda表达式构造的闭包，std::function会在其众多模版中实例其中一个来存储lambda表达式构造的闭包， 且std::function因为实现的不同，以及lambda表达式在构造闭包是捕获变量的多少，可能会在堆上分配空间来进行存储。但是auto变量却不在堆上分配空间，且可以肯定的是std::function分配的空间肯定多于auto分配的空间。介于实现的细节和内部的逻辑调用细节，通过std::function来调用lambda表达式构造的闭包几乎是肯定比通过auto来的慢。

Item 6: 当auto推断出不想要的类型时，请使用显式类型初始化习惯用法。

auto是通过变量的初始化来推导变量的数据类型，书中举了一个例子：

std::vector<bool> features(const Widget& w);

Widget w;
...
auto highPriority = features(w)[5];
...
processWidget(w, highPriority); //undefined behavior!

对于上面的这段代码，std::vector<bool>对[]操作返回的是std::vector<bool>::reference而不是通常的T&，即对bool类型的模板偏特化。对于这种auto推导出来的类型不是想要的类型的情况， 书中给的建议法是直接使用显示类型初始化：

auto highPriority = static_cast<bool>(features(w)[5]);
